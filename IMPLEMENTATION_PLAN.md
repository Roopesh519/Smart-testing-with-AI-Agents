### 2A — Add Global Token Rules to `SKILLS_CONTEXT.md`

**File:** `.claude/skills/SKILLS_CONTEXT.md`
**Where:** Add as the first section (top of file, before anything else)

```markdown
## Global Token Rules — All Skills Must Follow

These rules apply to EVERY skill, every run. Violating them is the single biggest
source of token waste. A 10-test run with browser_snapshot() per test = ~100k wasted tokens.

### Browser tool rules

| Operation | USE | NEVER USE | Why |
|-----------|-----|-----------|-----|
| Screenshots | `npx playwright screenshot --full-page URL file.png` | `browser_screenshot()` | base64 = 3k+ tokens per image |
| DOM reading | `browser_evaluate` with targeted CSS/JS | `browser_snapshot()` | full tree = 10k+ tokens |
| Login page | ONE `browser_snapshot()` allowed to find login selectors | `browser_snapshot()` anywhere else | one-time cost justified |
| Session reuse | `--storage-state .playwright-session.json` for all CLI calls | Re-logging in each run | saves login round-trip |

### Context output rules

| Content | Output to | Never output to |
|---------|-----------|-----------------|
| Full test reports | File (`outputs/`) | Chat |
| Full bug descriptions | Jira via MCP | Chat |
| Gherkin feature files | Write tool | Pasted in full to chat |
| Step definitions | Write tool | Pasted in full to chat |
| POM class | Write tool | Pasted in full to chat |
| Token tracking output | Suppressed | Chat (never show) |

**Chat shows only:** Phase announcements (1 line each), test count running total,
failure IDs, bug keys, final summary table. Nothing else.

### Static file load rules

| File | Load | Frequency |
|------|------|-----------|
| `BDD_TEMPLATES.md` | Once at Phase 1 start | Never reload in same run |
| `LOCATOR_PATTERNS.md` | Once at Phase 2 start | Never reload |
| `WCAG_CHECKS.md` | Once at skill start | Never reload |
| `test-charter.md` | Once when charter starts | Never reload |
| `context.md` | Once at qa-agent Step 0 | Never reload |

After loading, reference rules by name only (e.g. "per Locator Priority Rule 1").
**Never re-quote static content** in follow-up messages — it already exists in context.

### Prompt structure for cache efficiency

Structure every skill invocation in this order:
```
[1] STABLE BLOCK (cache-eligible — changes rarely or never)
    - Product context (context.md contents)
    - Skill instructions (SKILL.md)
    - Static templates and rule files loaded at start

[2] VARIABLE BLOCK (changes every run — goes last)
    - Card ID and fetched card data
    - Test results, bug details, current session data
```

The stable block qualifies for Anthropic's prompt cache when it exceeds 1024 tokens.
Cache TTL is 5 minutes. Keep the stable block identical across runs for the same product.
```

| Operation | Allowed | Forbidden | Token cost |
|-----------|---------|-----------|------------|
| Login page selectors | ONE `browser_snapshot()` at login URL only | `browser_snapshot()` on any other page | ~10k |

**After the ONE allowed login snapshot — never call `browser_snapshot()` again in this run.**
Use `browser_evaluate` with a specific CSS selector or JS expression for all DOM inspection.
